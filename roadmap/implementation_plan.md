## Overview

We're building a **Continuous Identity Trust & Adaptive MFA** web prototype that:
1. Captures behavioral biometrics (typing rhythm, mouse movement) in real-time
2. Computes a live Trust Score from multiple signals
3. Triggers adaptive MFA challenges when trust drops
4. Visualizes fraud rings using Neo4j graph database

### Step 2.1 — Keystroke Dynamics Collector

This is the **core innovation** — a JS module that silently captures:

```javascript
// What we capture per keystroke:
{
    key: "a",
    keyDown: 1718712345123,    // timestamp of key press
    keyUp: 1718712345189,      // timestamp of key release
    dwellTime: 66,             // keyUp - keyDown (ms)
    flightTime: 123,           // time since last keyUp (ms)
}
```

**How it works:**
- Attaches `keydown` and `keyup` listeners to the document
- Calculates dwell time (how long key is held) and flight time (gap between keys)
- Every 3 seconds, computes aggregate metrics and sends to backend:
  - Average dwell time
  - Average flight time
  - Typing speed (chars/sec)
  - Standard deviation of both timings

### Step 2.2 — Mouse Movement Tracker

```javascript
// What we capture per mouse event:

}
```

**Signals extracted:**
- Average mouse speed
- Path straightness (bots move in perfectly straight lines)
- Click frequency and location entropy
- Scroll behavior

## Phase 3: Trust Score Engine (The Brain)
**⏱️ Time: 2 hours**

### Step 3.1 — Behavioral Score (40% of total)

**Practical approach — NO ML TRAINING NEEDED for prototype:**

```python
# Phase 1: Baseline Building (first 30 seconds of any session)
# Collect the user's "normal" typing rhythm during the enrollment/login phase
# Store as their behavioral baseline in SQLite

# Phase 2: Continuous Comparison
# Every 3-second telemetry packet is compared against the baseline

def compute_behavioral_score(current_metrics, baseline):
    """
    Compare current typing/mouse behavior against stored baseline.
    Returns 0-100 score. 100 = perfectly matches baseline.
    """
    score = 100.0
    
    # Keystroke flight time deviation
    if baseline.avg_flight_time > 0:
        flight_deviation = abs(current.avg_flight_time - baseline.avg_flight_time)
        flight_z_score = flight_deviation / max(baseline.std_flight_time, 1)
        score -= min(flight_z_score * 10, 30)  # Max 30 point penalty
    
    # Keystroke dwell time deviation
    if baseline.avg_dwell_time > 0:
        dwell_deviation = abs(current.avg_dwell_time - baseline.avg_dwell_time)
        dwell_z_score = dwell_deviation / max(baseline.std_dwell_time, 1)
        score -= min(dwell_z_score * 10, 25)  # Max 25 point penalty
    
    # Typing speed deviation (>30% change = suspicious)
    speed_ratio = current.typing_speed / max(baseline.avg_typing_speed, 0.1)
    if speed_ratio > 1.3 or speed_ratio < 0.7:
        score -= 20
    
    # Mouse linearity (bots have very straight paths)
    if current.mouse_straightness > 0.95:  # Too straight = bot-like
        score -= 15
    
    return max(score, 0)
```

> [!TIP]
> **Why this works for the demo:** When YOU type and use the mouse, the baseline matches → score stays ~90-100. When your TEAMMATE types, their rhythm is different → score drops to ~50-65. This is the "wow" moment.

### Step 3.2 — Device Score (20% of total)

```python
def compute_device_score(current_fingerprint, known_fingerprints):
    score = 0
    
    if current_fingerprint.hash in known_fingerprints:
        score += 50  # Known device
    
    if current_fingerprint.timezone == "Asia/Kolkata":
        score += 15  # Expected timezone
    
    if current_fingerprint.platform in user_platforms:
        score += 20  # Known OS
    
    if current_fingerprint.screen_res in user_screens:
        score += 15  # Known screen
    
    return min(score, 100)
```

### Step 3.3 — Session Context Score (20% of total)

```python
def compute_session_score(session_data):
    score = 100
    
    # Time of day (banking hours = expected)
    hour = datetime.now().hour
    if 9 <= hour <= 21:
        score += 0  # Normal hours
    else:
        score -= 25  # Unusual hours
    
    # Request frequency (sudden burst = suspicious)
    if session_data.requests_per_minute > 30:
        score -= 30  # Possible bot
    
    # Session age (very long sessions = suspicious)
    if session_data.duration_minutes > 60:
        score -= 15
    
    return max(score, 0)
```

### Step 3.4 — Graph Risk Score (20% of total)

```python
def compute_graph_score(user_id, neo4j_driver):
    """
    Query Neo4j to check if this user's device/IP overlaps
    with known flagged accounts.
    """
    query = """
    MATCH (u:User {id: $user_id})-[:USES_DEVICE]->(d:Device)
          <-[:USES_DEVICE]-(other:User {flagged: true})
    RETURN count(other) as shared_flagged_devices
    """
    result = neo4j_driver.execute(query, user_id=user_id)
    
    if result.shared_flagged_devices > 0:
        return 20  # High risk — shares device with flagged accounts
    
    # Check IP overlap
    ip_query = """
    MATCH (u:User {id: $user_id})-[:CONNECTS_FROM]->(ip:IP)
          <-[:CONNECTS_FROM]-(other:User {flagged: true})
    RETURN count(other) as shared_flagged_ips
    """
    ip_result = neo4j_driver.execute(ip_query, user_id=user_id)
    
    if ip_result.shared_flagged_ips > 0:
        return 30
    
    return 100  # Clean — no fraud associations
```

### Step 3.5 — Composite Score & Decision Gateway

```python
def compute_trust_score(behavioral, device, session, graph):
    composite = (
        0.40 * behavioral +
        0.20 * device +
        0.20 * session +
        0.20 * graph
    )
    
    if composite > 85:
        return composite, "ALLOW"
    elif composite > 55:
        return composite, "STEP_UP"
    else:
        return composite, "BLOCK"
```

## Phase 4: Adaptive MFA System
**⏱️ Time: 1.5 hours**

### Step 4.1 — TOTP (Time-Based One-Time Password)

**How it works end-to-end:**

1. **Enrollment (during registration):**
   - Backend generates a TOTP secret using `pyotp.random_base32()`
   - Generates a QR code with `pyotp.totp.TOTP(secret).provisioning_uri()`
   - User scans QR code with Google Authenticator / Microsoft Authenticator
   - Secret is stored in the `users` table

2. **Verification (during step-up):**
   - User enters 6-digit code from their authenticator app
   - Backend verifies with `pyotp.TOTP(secret).verify(code)`
   - If valid → trust score recovers, session continues
   - If invalid → session blocked

### Step 4.2 — Face Capture Challenge (Simulated Liveness)

**What we actually build:**
1. When STEP_UP is triggered, the UI shows a webcam modal
2. Instructions: *"Look at the camera, then slowly turn your head left"*
3. We capture 2-3 frames using the WebRTC `getUserMedia` API
4. For the prototype, we **simulate** verification:
   - If a face is detected (using a simple canvas pixel analysis or just a timer), → pass
   - The point is the **UX flow**, not real face matching
5. In the report, explain this would use a real liveness API (AWS Rekognition / FaceTec) in production

> [!NOTE]
> For the judges, the impressive part is the **adaptive triggering** — not whether face matching is real. Showing that the system *chose* face liveness because OTP is useless (thief has the phone) is the insight.

### Step 4.3 — Adaptive MFA Selection Logic

```python
def select_mfa_method(trust_context):
    """
    Choose the right MFA based on WHY trust dropped.
    This is the 'adaptive' part that makes Sentinel special.
    """
    if trust_context.device_changed:
        # New device — user probably still has their phone
        return "TOTP"  # Authenticator app code
    
    if trust_context.behavioral_anomaly:
        # Different person typing — could be device snatch
        return "FACE_LIVENESS"  # Need biometric, OTP useless
    
    if trust_context.geo_anomaly:
        # Remote access from unusual location
        return "TOTP"  # Standard re-auth
    
    if trust_context.admin_sop_violation:
        # Insider threat — need supervisor approval
        return "MANAGER_APPROVAL"  # Show in SOC dashboard
    
    return "TOTP"  # Default fallback
```

### 7.1 — Behavioral Biometrics Model (What to Tell Judges)

**Algorithm:** Isolation Forest (unsupervised anomaly detection)

**Why Isolation Forest:**
- No labeled "fraud" data needed — learns what's "normal" and flags anomalies
- Fast inference (~1ms) — suitable for real-time scoring
- Available in scikit-learn — `from sklearn.ensemble import IsolationForest`

**Training approach:**
```python
from sklearn.ensemble import IsolationForest

# Features per telemetry window (3-second intervals)
features = [
    'avg_flight_time',      # ms between key releases
    'std_flight_time',      # consistency of timing
    'avg_dwell_time',       # ms key held
    'std_dwell_time',       # consistency
    'typing_speed',         # chars per second
    'mouse_avg_velocity',   # px/sec
    'mouse_straightness',   # 0-1, how linear the path is
    'click_frequency',      # clicks per second
]

# Train on 1 user's "normal" behavior (collected during enrollment)
model = IsolationForest(
    contamination=0.1,      # Expect 10% anomalies
    n_estimators=100,
    random_state=42
)
model.fit(user_normal_data[features])

# Predict: -1 = anomaly (different user), 1 = normal (same user)
prediction = model.predict(new_telemetry[features])
anomaly_score = model.decision_function(new_telemetry[features])
```

**Datasets to reference:**
- Clarkson University Keystroke Dataset (continuous typing dynamics)
- Balabit Mouse Dynamics Challenge (mouse behavior identification)
